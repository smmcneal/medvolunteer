I'm building a medical volunteer management platform. It has three parts:
1. A Progressive Web App (Next.js, volunteer-facing, mobile-first, installable)
2. A web admin dashboard (Next.js 14, App Router, TypeScript, Tailwind)
3. A Supabase backend (Postgres, Auth, Storage, Edge Functions)

Both the volunteer PWA and the admin dashboard live in the same Next.js
project — volunteer routes under /app/(volunteer)/, admin under
/app/dashboard/. They share types, Supabase client config, and edge
functions but have completely separate layouts and auth flows.

VOLUNTEER PWA — VOLUNTEER-FACING
═══════════════════════════════════════════════════════

Tech: Next.js 14 (App Router), TypeScript, Tailwind CSS,
      Supabase JS SDK (@supabase/ssr), next-pwa (Workbox),
      Web Push API (VAPID), Geolocation API

Routes live under /app/(volunteer)/ with a shared mobile-first
layout: bottom tab bar, safe-area insets, touch targets ≥ 44px.
Installable via Web App Manifest (manifest.json).

Tab 1: Home  (/volunteer/home)
  - Greeting with volunteer name + status badge
  - Next upcoming shift card (tap to view details)
  - Onboarding checklist progress bar (if not fully onboarded)
  - Credential expiry warnings (badges for credentials
    expiring within 30 days)
  - Quick clock-in button if a shift is active now

Tab 2: My Shifts  (/volunteer/shifts)
  - Upcoming shifts list with location + time
  - Manual clock in/out button (always visible as fallback)
  - Past shifts list with hours logged
  - Shift detail modal: role, location map link, co-volunteers

Tab 3: Learn  (/volunteer/learn)
  - Learning modules list with progress rings
  - Module detail: ordered lesson list, locked indicator
    if prerequisite modules are incomplete
  - Lesson player:
      video → <iframe> embed (YouTube / Vimeo URL)
      text  → markdown-rendered content
      quiz  → one question at a time, immediate feedback,
               score shown on completion
  - Module completion badge on finish

Tab 4: My Profile  (/volunteer/profile)
  - Personal info display (read-only; edit via admin)
  - Credentials list with expiry badges
  - Documents list (tap to open signed PDF in new tab)
  - Background check status card
  - Log out button

Tab 5: Messages  (/volunteer/messages)
  - Inbox: messages sent through the platform
  - Unread count badge on tab icon
  - Tap message → full detail view (marks as read)
  - Channel badge (email / SMS / push) on each message



WEB PUSH NOTIFICATIONS (VAPID):
  - Request permission on first login
  - Store PushSubscription (endpoint, p256dh, auth) in
    push_subscriptions table
  - Triggers: shift reminder 30 min before start,
    clock-in/out confirmation, onboarding stage assigned,
    new message received

PWA INSTALL:
  - manifest.json: name, short_name, icons (192/512px),
    theme_color (#1B2A4A), background_color (#ffffff),
    display: standalone, start_url: /volunteer/home
  - beforeinstallprompt: show "Add to Home Screen" banner
    on first visit after 30 seconds
  - Service Worker (Workbox via next-pwa):
    cache-first for static assets + API GET responses
    network-first for Supabase data fetches
    offline fallback page for no-network state

═══════════════════════════════════════════════════════
SUPABASE EDGE FUNCTIONS
═══════════════════════════════════════════════════════

clock-in         POST: volunteer_id, location_id, method
                 → insert time_entry, send push notification

clock-out        POST: volunteer_id, time_entry_id
                 → update time_entry, compute duration

send-message     POST: message_id
                 → route to Resend (email) or Twilio (SMS)
                 → send Web Push to push_subscriptions

send-push        POST: volunteer_id, title, body, url
                 → look up push_subscriptions, send via
                   Web Push Protocol (VAPID)

initiate-bgcheck POST: volunteer_id
                 → call Checkr API, store external_id

bgcheck-webhook  POST (Checkr webhook)
                 → update background_checks status + result

docusign-webhook POST (DocuSign webhook)
                 → update documents.status, signed_at

advance-onboarding POST: volunteer_id, stage_id
                 → mark stage complete, check if all stages
                   done → update volunteer status to 'active',
                   trigger welcome email + push notification

DOMAIN NAME SUGGESTIONS
═══════════════════════════════════════════════════════

Recommendation: Use a .app TLD — browsers enforce HTTPS
on .app domains, which is required for geolocation and
Web Push anyway. It also signals "this is a product."

TIER 1 — Best fits (clear meaning, memorable, available
          on most registrars ~$15–20/yr):

  careforce.app       — mission-driven, team feel,
                        works for any med volunteer org
  medshift.app        — exactly describes the core
                        function (shifts + medical)
  medvol.app          — short form of "MedVolunteer",
                        easy to type on mobile
  volunteermd.org     — .org suits nonprofits/hospitals;
                        MD signals medical credibility
  careops.app         — "operations" implies the
                        admin + scheduling angle

TIER 2 — Strong alternates:

  medcrew.app         — team-oriented, easy to remember
  shiftmd.app         — shift-first naming
  medready.app        — implies preparedness / compliance
  carestaff.app       — straightforward, professional
  volunteerhq.health  — .health TLD adds credibility;
                        "HQ" implies the admin dashboard

TIER 3 — Short / brandable (check availability):

  volmd.app           — ultra-short
  crewmd.app          — concise
  myvol.health        — friendly, volunteer-facing feel

WHAT TO AVOID:
  - medvolunteer.com  — likely taken or expensive
  - anything with "platform" or "pro" — generic
  - hyphens           — harder to type on mobile

RECOMMENDATION:
  careforce.app or medshift.app — both are short,
  HTTPS-enforced, professional, and clearly describe
  a medical volunteer scheduling tool.

CONSTRAINTS
═══════════════════════════════════════════════════════
- All integrations (Checkr, Twilio, etc.) should be
  stubbed with a clean interface during build so the
  app works end-to-end with mock data. Wire real keys last.
- Use Supabase RLS (Row Level Security) from day one —
  never skip this
- Volunteer PWA targets modern browsers (Chrome 90+,
  Safari 16.4+). Geolocation + Web Push require HTTPS
  in production; use localhost for dev.
- iOS Safari 16.4+ supports Web Push only when the PWA
  is installed to the Home Screen. Show a clear install
  prompt for iOS users.
- Store no PHI in plain text; treat all volunteer data
  as sensitive
- Ask before making assumptions about any ambiguous
  business rule
