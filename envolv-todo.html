The first successful deploy since 2026-04-29 shipped without NEXT_PUBLIC_SUPABASE_URL/ANON_KEY — not because they were missing from Vercel, but because NEXT_PUBLIC_* inlines at BUILD TIME and the vars were saved (17:53 UTC) ~10h AFTER the last build (07:17 UTC).   • 🛑 Cowork sandbox — re-diagnosed 2026-07-12, and it's worse than we thought. Not staleness: the Linux mount serves a truncated prefix of each file — byte-accurate from the start, then it just stops. envolv-todo.html came back as 34,876 bytes ending mid-word, no closing </html>, while still reflecting edits made seconds earlier. Fresh content, cut short. Why that's dangerous: git compares the truncated files against intact HEAD blobs and reads the missing tails as deletions — a sandbox git diff showed 898 phantom deletions across 18 files. A git add -A && git commit from the sandbox would commit those truncations for real, amputating the back half of ShiftsView.tsx and 17 others in a commit that looks perfectly clean. Never run git/npm/lint/tsc through the sandbox for this repo; treat any sandbox git status/git diff as fiction. Claude's Read/Write/Edit tools hit Windows directly and are accurate; build and commit on Windows.
