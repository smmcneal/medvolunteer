# Pricing-Page Feature Prompts

Every feature promised on the Envolv pricing page that does **not** exist in this repo yet, as a ready-to-use build prompt. Paste a prompt into the Notion Feature Requests DB `Component/File` field (the AB-### pipeline treats that field as the entire spec — never leave it blank) or run it directly in Claude Code.

**Audit date: 2026-07-12.** Already built, no prompt needed: volunteer self-service portal, mobile app access (PWA), scheduling & hour tracking, standard reporting + CSV export, email notifications, advanced scheduling & approvals (recurring shifts, hour approval flow), logo branding (org `logo_url` in Settings), geofenced clock-in (volunteer `volunteerClockIn` with `method: 'geofence'`), multi-location within one org.

Prompts are ordered by dependency — 1 and 2 unblock most of the rest. Every prompt assumes repo conventions from CLAUDE.md: server actions start with `requireAdmin()`/`requireVolunteer()`, all DB access via `createAdminClient()`, page.tsx fetches / *View.tsx renders, new tables get deny-by-default RLS, all UI strings through `useAdminT()`/`useT()`, migrations via `npx supabase migration new` then regenerate `web/types/database.ts`.

## 1. Plan tiers, limits, and feature gating

> Add subscription plan awareness to the platform. Migration: add `plan` (`starter` | `professional` | `scale` | `enterprise`, default `starter`), `plan_limits` jsonb, and `billing_status` to the `organizations` table. Create `web/lib/plan.ts` exporting a `PLAN_LIMITS` map — starter: 100 volunteers / 2 admin seats / 1 industry template / 0 SMS; professional: 1,000 / 5 / 3 templates / 500 SMS/mo; scale: 5,000 / 15 / all templates / 2,500 SMS/mo; enterprise: unlimited everything — plus helpers `getOrgPlan()`, `checkVolunteerCap()`, `checkAdminSeatCap()`, and `hasFeature(feature)` for boolean gates (`custom_reports`, `sms`, `background_checks`, `kiosk`, `api_access`, `advanced_permissions`, `white_label`, `sso`). Enforce caps where rows are created: volunteer creation/invite actions in `dashboard/volunteers/actions.ts` and the public `/apply` flow (volunteer cap), admin creation (seat cap) — throw a clear, translated error naming the plan and limit, with an upgrade hint. Add a read-only plan summary card to `/dashboard/settings` showing current plan, usage vs. caps (volunteers, admin seats, SMS used this month). Do NOT build payment processing in this task. Gate nothing that already works for existing orgs — default existing org rows to `enterprise` in the migration so nothing breaks at cutover.

## 2. Stripe billing & subscription management

> Integrate Stripe subscriptions for the four plans (Starter $29, Professional $79, Scale $149 monthly; annual prices at ~17% discount; Enterprise is sales-led, no self-serve checkout). Add `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` to `.env.example`. Migration: `billing_customers` table (org_id, stripe_customer_id, stripe_subscription_id, current_period_end, status), service-role-only RLS. Build `/dashboard/settings/billing`: current plan, monthly/annual toggle, upgrade/downgrade via Stripe Checkout session (server action, `requireAdmin()` first line), and a "Manage billing" button opening the Stripe customer portal. Webhook route `POST /api/webhooks/stripe` (signature-verified, like the existing Vercel webhook in `app/api/webhooks/vercel/route.ts`): on subscription created/updated/deleted, update `organizations.plan` and `billing_status`. Grace period: past_due keeps access 14 days, then admin dashboard shows a payment banner but data stays intact — never delete data on lapse. Depends on prompt 1's plan column.

## 3. Industry templates (1 / 3 / all by tier)

> Build an industry template system so a new org can start pre-configured for its sector instead of the current medical-clinic-only defaults. A template = a named bundle of seed rows: volunteer categories + descriptions, pipeline/checklist items, document requirement rules, shift role names, message templates, and auto-message rules. Create `web/lib/templates/` with one file per template exporting a typed `IndustryTemplate` object; ship at least: Medical Clinic (extract from current `supabase/seed.sql` defaults), Food Bank / Pantry, Animal Shelter, Events & Festivals, Education / Tutoring, Faith-Based. Add an "Apply industry template" section in `/dashboard/settings` (admin action, idempotent — inserts only rows that don't already exist by name, never deletes). Gate by plan via `hasFeature`/`PLAN_LIMITS` from `web/lib/plan.ts`: starter picks 1 template, professional up to 3, scale/enterprise all. Record applied templates in a new `org_templates` table (migration, deny-by-default RLS). All template display names/descriptions through admin i18n (EN + ES).

## 4. SMS sending integration + monthly quota

> Wire up real SMS delivery. The `sms` channel already exists in the messages UI and schema (`dashboard/messages/actions.ts` — see the "sms/push channels are not wired yet" comment) but nothing sends. Integrate Twilio: `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, `TWILIO_FROM_NUMBER` env vars (`.env.example` + document Vercel setup), new `web/lib/sms.ts` mirroring the shape of `web/lib/email.ts`, silently no-op when env vars are absent (same pattern as Notion automation). Dispatch SMS in `sendMessage` for immediate sends and in `/api/cron/send-auto-messages` for scheduled ones, respecting the existing status flow (`draft`→`sent`/`failed`, scheduled pass stays idempotent). Add per-org monthly quota metering: `sms_usage` table (org_id, month, count) with service-role-only RLS; check against `PLAN_LIMITS` sms cap before dispatch, fail the send with a translated quota error when exceeded, and show usage on the settings plan card. Volunteers need a `phone_sms_opt_in` boolean (migration) — only send to opted-in numbers, and add the opt-in toggle to the volunteer profile tab.

## 5. Background check provider integration

> Integrate a background check provider (Checkr) behind an abstraction so others can follow. Background checks are currently a manually-tracked pipeline step (`VolunteerTabs.tsx` background_check tab). Add `web/lib/background-check.ts` with a provider interface (`orderCheck`, `getStatus`, webhook parsing) and a Checkr implementation using `CHECKR_API_KEY` (no-op when unset). Migration: add `provider`, `provider_check_id`, `provider_status`, `report_url` to the existing background check data model. In the volunteer detail Background Check tab, add an admin "Order background check" action (`requireAdmin()`, consent must already be on file — block otherwise) and status auto-refresh via `POST /api/webhooks/checkr` (signature-verified). Map provider statuses onto the existing manual statuses so manual entry keeps working for orgs without the integration. Gate with `hasFeature('background_checks')` (professional+). Surface errors in the existing `run()` error pattern.

## 6. Kiosk mode

> Build a shared-device kiosk for on-site volunteer check-in/out, complementing the existing geofenced clock-in (`volunteer/(tabs)/shifts/actions.ts`). New route zone `/kiosk`: an admin action in Settings generates a kiosk session for a specific location (new `kiosk_sessions` table: org_id, location_id, token hash, expires_at; deny-by-default RLS), producing a URL + QR code to open on a tablet. The kiosk page is fullscreen, large-touch-target, no volunteer login: volunteer searches their name or enters a 4-digit PIN (add `kiosk_pin` to volunteers, settable from their profile), sees today's shifts at that location, taps to clock in/out. Server actions validate the kiosk token server-side and record time entries with a new `method: 'kiosk'` (extend the existing `'manual' | 'geofence'` union in types and clock-in actions). Auto-logout to the search screen after 15s idle. Session revocation list in Settings. Gate with `hasFeature('kiosk')` (scale+). EN + ES via the volunteer i18n context.

## 7. Public API + data exports

> Build a public REST API. Migration: `api_keys` table (org_id, name, key hash, last_used_at, revoked_at; service-role-only RLS). Settings gains an "API" section (gated `hasFeature('api_access')`, scale+): create key (show plaintext once), revoke, last-used. Routes under `/api/v1/`: `GET /volunteers`, `GET /shifts`, `GET /hours`, `GET /messages` with pagination, plus `GET /export/{volunteers|hours|shifts}.csv` full-dataset exports (the CSV logic exists client-side in `ReportsView.tsx` `downloadCSV` — reimplement server-side, don't try to share it). Every route authenticates via `Authorization: Bearer <key>` against the hash, scopes all queries to the key's org_id, 404s cleanly, and rate-limits (60 req/min per key, in-memory is fine for now). Never expose service-role Supabase access; the API layer uses `createAdminClient()` internally after key validation, mirroring how auth guards precede data access everywhere else. Add `docs/api.md` documenting every endpoint with curl examples.

## 8. Advanced permissions (roles)

> Replace the binary `admin_users` model with roles, without breaking existing guards. Migration: add `role` (`owner` | `admin` | `coordinator` | `viewer`, default `admin` for existing rows) and optional `location_ids uuid[]` scoping to `admin_users`. Extend `web/lib/auth.ts`: keep `requireAdmin()` behavior identical (any role passes — audit call sites to confirm nothing breaks), add `requireRole(minRole)` and `getAdminRole()`. Enforcement: owner-only — billing, plan changes, admin seat management, org profile; admin — everything current; coordinator — shifts, hours approval, messages, volunteers (read/write) but no Settings; viewer — read-only dashboard (hide/disable mutation UI in Views AND guard the server actions — layout checks alone are not protection, per CLAUDE.md). Location-scoped coordinators only see shifts/volunteers for their locations. Settings gains an "Admins & roles" manager (owner-only). Gate role management UI with `hasFeature('advanced_permissions')` (scale+); below that plan every admin row behaves as `admin`. Seat cap from prompt 1 applies on admin creation.

## 9. Full white-label branding

> Extend the existing logo branding (org `logo_url` in Settings) to full white-label. Migration: add `brand_primary_color`, `brand_accent_color`, `brand_app_name`, `custom_domain`, `email_from_name`, `hide_platform_branding` to `organizations`. Apply: CSS variables injected in `DashboardShell.tsx` and `VolunteerShell.tsx` from org branding (fall back to current palette); volunteer PWA manifest served dynamically (new `app/manifest.ts` reading org branding — name, icons, theme color) and service worker cache name bumped; transactional emails in `web/lib/email.ts` use `email_from_name` and org logo in the template; "Powered by Envolv" footer rendered everywhere by default and hidden when `hide_platform_branding` (scale+ via `hasFeature('white_label')`). Custom domain: store + verify via a TXT record check (server action), document the Vercel domain-attach step in `docs/white-label.md` — actual domain routing is manual ops for now. Branding editor lives in Settings with live preview, owner/admin only. All new strings i18n'd.

## 10. Multi-org & network dashboard (unlimited sites & orgs)

> Support multiple organizations per deployment for networks/agencies. Today org resolution is hardcoded single-org (`select … from organizations limit 1 single()` — grep for it and centralize into `web/lib/org.ts` FIRST as its own commit). Migration: `org_memberships` (user_id, org_id, role) replacing the implicit single-org assumption for admins; volunteers stay org-scoped via their existing org_id. Admin UI: org switcher in the `Sidebar.tsx` header (only renders with >1 membership); all dashboard queries filter by the active org (cookie-persisted selection, validated server-side against membership on every action — never trust the cookie alone). New `/dashboard/network` page (visible to users with 2+ org memberships): cross-org rollup of volunteer counts, hours this month, upcoming shifts, per-org drill-in links. Keep RLS deny-by-default; all reads still flow through `createAdminClient()` after guards that now also verify org membership. This is the largest prompt here — split into (a) org resolution refactor, (b) memberships + switcher, (c) network dashboard as three Notion tasks if the nightly agent struggles.

## 11. SSO / SAML

> Add enterprise SSO. Supabase Auth supports SAML 2.0 SSO on Pro+ — use it rather than hand-rolling. Migration/config: `sso_connections` table (org_id, provider label, supabase sso domain, enforced boolean). Settings (enterprise only, `hasFeature('sso')`, owner role): register an SSO domain (calls Supabase admin API to create the SSO provider with the org's IdP metadata URL), toggle "require SSO" which blocks password login for emails on that domain. Login pages (`/login` and `/volunteer/login`): detect email domain on submit; if an enforced SSO connection matches, redirect into `supabase.auth.signInWithSSO({ domain })` instead of password auth. `/auth/callback` already handles the return leg — verify it passes through SSO sessions and lands admins vs. volunteers correctly via the existing layout guards. Document IdP setup (Okta + Entra ID walkthroughs) in `docs/sso.md`. Fail closed: if the SSO provider errors, show a translated error, never fall back to password when SSO is enforced.

## 12. Custom integrations & data migration tooling

> Build the migration/integration toolkit promised to Enterprise. Three parts. (a) CSV volunteer import: Settings → "Import volunteers" wizard (upload CSV, map columns to volunteer fields, preview with per-row validation errors, dry-run then commit; dedupe by email; imported rows get status `prospect`). (b) Competitor importers: preset column mappings for common exports (at minimum Timecounts, SignUpGenius, Galaxy Digital), selectable in the same wizard. (c) Outbound webhooks: `webhook_endpoints` table (org_id, url, secret, events[]; deny-by-default RLS), Settings CRUD, and dispatch (HMAC-signed, same signing pattern as the inbound Vercel webhook) on events: volunteer.created, volunteer.status_changed, shift.created, hours.approved, message.sent — fire-and-forget with 3 retries, log to a `webhook_deliveries` table, surface recent deliveries + redeliver button in Settings. Import is available to all plans (it's an onboarding tool); outbound webhooks gate with `hasFeature('api_access')`.

## 13. In-app support widget + tiered routing

> Add a support entry point that fulfills the tiered support promises (Starter: email 48hr; Professional: email 24hr + chat; Scale: priority chat + phone; Enterprise: dedicated CSM). In-app: a "Help & Support" item in `Sidebar.tsx` opening a panel with (a) links to docs, (b) a contact form that emails support via `web/lib/email.ts` with org, plan, user, and current page prefilled, and (c) plan-aware messaging — show the SLA for the org's plan, show chat/phone options only on qualifying plans (chat can be a mailto/Crisp placeholder until a chat vendor is chosen; make the vendor pluggable behind `web/lib/support.ts`). Tag outbound support emails with the plan so the inbox can be triaged by SLA. Volunteer PWA gets a simpler "Contact your coordinator" version. All strings i18n'd.

## 14. Support SLA & CSM playbook (document)

> Write `docs/support-playbook.md`: definitions of the four support tiers exactly as sold (48hr email / 24hr email + chat / priority chat + phone / dedicated CSM + SLA); response-time SLAs and business-hours definition; triage flow from the support inbox including how plan tags (from prompt 13) map to queues; escalation path; what "dedicated CSM" includes for Enterprise (onboarding call, quarterly review, named contact); template replies for the 10 most likely requests. Keep it honest to a one-person company: note which promises require hiring or tooling before they can be sold, so sales conversations don't outrun capacity.

## 15. Security review pack & DPA (documents)

> Produce the Enterprise trust package. (a) `docs/security-overview.md`: architecture summary (Next.js on Vercel, Supabase Postgres with deny-by-default RLS, service-role access only behind server-side auth guards), data flow diagram, encryption at rest/in transit, backup posture, subprocessor list (Vercel, Supabase, Resend, Twilio, Checkr, Stripe as adopted), vulnerability-handling statement, and honest gaps (no SOC 2 yet). (b) A DPA template as `docs/dpa-template.md` covering GDPR/CCPA-standard clauses: roles (controller/processor), processing purposes, subprocessors, breach notification windows, deletion on termination — clearly marked DRAFT, NOT LEGAL ADVICE, for attorney review before first use. (c) A security questionnaire crib sheet (`docs/security-questionnaire-answers.md`) answering the ~30 questions every hospital/government procurement asks, mapped to what the codebase actually does.

## 16. Data retention & deletion (supports DPA + trust)

> Implement the data-lifecycle guarantees the DPA will promise. Org deletion: owner-only, double-confirmed server action that hard-deletes all org data (volunteers, shifts, hours, messages, uploads in storage, push subscriptions) after a 30-day soft-delete grace window (`deleted_at` on organizations; a daily job — piggyback on the existing Vercel midnight cron in `/api/cron/send-auto-messages`'s pattern but as a separate `/api/cron/retention` route protected by `CRON_SECRET` — purges past-due orgs). Volunteer right-to-erasure: admin action on the volunteer detail page that anonymizes a volunteer (strip PII, keep aggregate hour totals for org reporting) — distinct from delete. Export-before-delete: org deletion flow offers a full JSON + CSV dump first (reuse prompt 7's export code). Log every deletion/anonymization to an `audit_log` table (service-role-only RLS). Document the retention schedule in `docs/security-overview.md`.
